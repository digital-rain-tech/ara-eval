## High

### 1. Incorrect benchmark classification in starter scenario data
- File path: `scenarios/starter-scenarios.json:83`
- Severity: high
- Category: Diff
- Description: `banking-customer-service-001` marks `regulatory_exposure` as `D` even though the scenario includes an address change and the note references PDPO handling. Under the rubric, that is at least general compliance exposure rather than "unregulated". This weakens the benchmark data and can bias the lab toward an unsafe autonomy recommendation.
- Suggested fix: Reclassify the scenario to match the rubric and add a validation check that flags rubric-inconsistent reference data.

### 2. Core decision logic is untested and concentrated in a single script
- File path: `labs/lab-01-risk-fingerprinting.py:149`
- Severity: high
- Category: Holistic
- Description: The repository's deterministic policy layer, including `apply_gating_rules`, `personality_delta`, and response parsing, lives in one script without visible automated tests. That makes refactors risky because readiness classifications can change silently.
- Suggested fix: Extract the decision logic into an importable module and add unit tests covering hard gates, soft gates, all-`C/D` readiness, malformed fingerprints, response parsing failures, and representative personality-delta cases.

## Medium

### 3. Missing agent guidance in major working directories
- File path: `.`
- Severity: medium
- Category: Holistic
- Description: The repo has a root `CLAUDE.md` but no root `AGENTS.md`, and the main directories `docs/`, `labs/`, and `scenarios/` do not contain directory-local agent instructions or Claude-compatible symlinks. That leaves editing conventions, safe boundaries, and validation steps implicit instead of discoverable where work happens.
- Suggested fix: Add a root `AGENTS.md`, symlink `CLAUDE.md` to it for compatibility, and add scoped `AGENTS.md` files in `docs/`, `labs/`, and `scenarios/` documenting purpose, conventions, validation commands, and common pitfalls.

### 4. LLM response handling is brittle and can abort the entire run
- File path: `labs/lab-01-risk-fingerprinting.py:197`
- Severity: medium
- Category: Diff
- Description: The evaluation path assumes `response.content[0].text` exists and that the model always returns parseable JSON. A malformed response, SDK shape change, or transient API error will terminate the batch instead of isolating the failure to one scenario.
- Suggested fix: Validate the response structure, catch JSON parse failures and API exceptions, and record per-scenario error states so one bad response does not stop the full run.

### 5. Startup configuration is not validated and `.env` support is incomplete
- File path: `labs/lab-01-risk-fingerprinting.py:9`
- Severity: medium
- Category: Diff
- Description: The docs mention `python-dotenv`, but the script does not load `.env` and does not verify that `ANTHROPIC_API_KEY`, input data, or output paths are valid before running. Failures therefore occur late and with poor diagnostics.
- Suggested fix: Either remove `python-dotenv` from the documented setup or call `load_dotenv()`, then add explicit startup validation for required environment variables, scenario schema, and output directory availability.

### 6. Project setup is not reproducible from a dependency manifest
- File path: `README.md:13`
- Severity: medium
- Category: Holistic
- Description: The repository documents manual `pip install` commands but does not provide a `requirements.txt`, `pyproject.toml`, lockfile strategy, or declared Python version. That makes environments drift over time and weakens agent readiness because setup assumptions are not machine-readable.
- Suggested fix: Add `pyproject.toml` or `requirements.txt`, define the supported Python version, and update the docs to use the manifest instead of ad hoc package installation.

### 7. Main orchestration mixes unrelated responsibilities
- File path: `labs/lab-01-risk-fingerprinting.py:303`
- Severity: medium
- Category: Holistic
- Description: `main()` combines file loading, client setup, network execution, business logic, console rendering, and persistence in one flow. That structure is difficult to test and will become harder to maintain as the lab grows.
- Suggested fix: Split the script into focused modules such as `config`, `evaluator`, `gating`, `reporting`, and `cli`, with `main()` reduced to orchestration only.

### 8. External model integration is not abstracted behind a stable interface
- File path: `labs/lab-01-risk-fingerprinting.py:197`
- Severity: medium
- Category: Holistic
- Description: Anthropic API usage is embedded directly in the evaluation logic. Retries, provider swaps, mocking, timeout policy, and response normalization would all require invasive changes.
- Suggested fix: Introduce a small adapter or client interface that owns request construction, retries, timeout handling, and response normalization.

### 9. Error handling is inconsistent across I/O and model execution paths
- File path: `labs/lab-01-risk-fingerprinting.py:197`
- Severity: medium
- Category: Holistic
- Description: File I/O failures, missing scenario keys, API errors, rate limits, and invalid model output appear to propagate as raw exceptions. That makes the tool brittle for repeated use and difficult for agents to operate safely.
- Suggested fix: Add structured exception handling around file loading, model calls, and JSON parsing with actionable user-facing errors and retry logic for transient failures.
