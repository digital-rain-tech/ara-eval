# Consolidated Review

## High

### 1. Unauthenticated inspector endpoints expose stored prompts, model outputs, and chat history
File: `web/src/app/api/requests/route.ts`, `web/src/app/api/runs/route.ts`, `web/src/app/api/chat/sessions/route.ts`
Line: `4-27`, `4-20`, `4-23`
Severity: high
Category: Diff
Description: The new inspector/history endpoints return full request rows, run metadata, and chat transcripts with no authentication or authorization checks. Those rows include `raw_request`, `raw_response`, `parsed_result`, and chat content, so a deployed instance would expose evaluation prompts, model outputs, and session history to any caller who can hit the route.
Suggested fix: Put these routes behind authentication and explicit authorization before shipping. If they are intended for local-only use, enforce that in middleware or deployment config and avoid returning raw payload columns to unauthenticated clients.

### 2. Client-controlled model selection can spend server-side OpenRouter credits on arbitrary models
File: `web/src/app/api/evaluate/route.ts`, `web/src/app/api/chat/route.ts`
Line: `10-22`, `36`; `19-29`, `43`, `114-117`
Severity: high
Category: Diff
Description: Both write endpoints accept `model` from the request body and forward it directly to OpenRouter while using the server's API key. Because there is no auth gate or allowlist enforcement, an anonymous caller can choose any provider/model ID the account can access, including paid or untested models, and burn credits or trigger unexpected behavior.
Suggested fix: Enforce an allowlist from `shared/models.json` on the server, reject unknown model IDs with `400`, and require authenticated access before permitting any model override.

## Medium

### 3. Invalid prompt parameters crash with 500s instead of returning a controlled 4xx
File: `web/src/app/api/prompt/route.ts`, `web/src/app/api/chat/route.ts`, `web/src/lib/prompts.ts`
Line: `14-26`; `31-91`; `40-75`, `133-189`
Severity: medium
Category: Diff
Description: `personality`, `jurisdiction`, and `rubric` come from query/body params and are dereferenced without validation. Unknown values fall through to expressions like `jurisdictions[jurisdiction].file` and `personalities[personalityId].file`, which raises a runtime exception and surfaces a `500` for simple bad input.
Suggested fix: Validate all prompt selectors against the loaded indexes before calling the prompt builder, return `400` for unknown values, and add route tests covering invalid personality/jurisdiction/rubric combinations.

### 4. Web-run logging drops raw provider responses and never records total run duration
File: `web/src/app/api/evaluate/route.ts`, `web/src/lib/db.ts`
Line: `78-91`, `106-119`, `130-138`; `197-255`
Severity: medium
Category: Diff
Description: Successful and failed web evaluations always persist `rawResponse: null`, and the run update never sets `total_duration_ms`. That makes the new request inspector and run history incomplete by design: raw response panels stay empty and duration in the history UI remains `-` even when the run succeeded.
Suggested fix: Capture the actual OpenRouter request/response payloads in the evaluation path, persist them through `logRequest`, and compute/write `total_duration_ms` when finalizing the run.

### 5. The new web API/data layer has almost no behavioral test coverage
File: `web/src/__tests__/gating.test.ts`, `web/src/app/api`, `web/src/lib/db.ts`
Line: `1-52`, `n/a`, `n/a`
Severity: medium
Category: Diff
Description: The web test suite currently exercises only the shared gating fixtures. None of the newly added route handlers, prompt composition error paths, SQLite persistence helpers, or chat/session flows have automated coverage, so the main risks in this change set are untested.
Suggested fix: Add route-level tests for `/api/evaluate`, `/api/chat`, `/api/prompt`, `/api/requests`, and `/api/chat/sessions`, plus DB-focused tests for run/chat persistence and inspector queries.

### 6. Agent guidance is only defined at the repository root, not near major work areas
File: `AGENTS.md`, `ara_eval/`, `labs/`, `tests/`, `web/src/`
Line: `1-3`, `n/a`, `n/a`, `n/a`, `n/a`
Severity: medium
Category: Holistic
Description: The root `AGENTS.md` is only a pointer to `CLAUDE.md`, and there are no local `AGENTS.md`/`CLAUDE.md` files in the major code areas an agent would edit most often. That makes the project less navigable for agent-driven work and leaves directory-specific conventions, test expectations, and pitfalls undocumented where they matter.
Suggested fix: Add short local agent guides for `ara_eval/`, `labs/`, `tests/`, and `web/` or `web/src/`, and keep Claude-compatible links/symlinks in each major directory.

## Low

### 7. Environment loading is centralized but not validated fail-fast
File: `web/src/lib/env.ts`
Line: `1-15`
Severity: low
Category: Holistic
Description: The web app now centralizes dotenv loading, but it still does not validate required environment variables at startup. Missing values are only discovered later when a request path calls `getApiKey()`, which turns a deploy-time misconfiguration into a runtime failure.
Suggested fix: Add a small startup schema for required env vars in `web/src/lib/env.ts` and fail immediately with a clear error when required values are missing or malformed.
